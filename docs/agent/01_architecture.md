# Архитектура (Architecture)

## Системен преглед

"The Best Idea Eatery" е Single Page Application (SPA), което използва React на фронтенда и Firebase като Backend-as-a-Service (BaaS) за съхранение на данни и автентикация.
Приложението е мултиезично и е фокусирано върху управлението на кулинарни рецепти, наличности от продукти (килер) и общност за любители на кулинарията.

## Технологии

*   **Фронтенд:** React 19
*   **Сбор (Build tool):** Vite
*   **Маршрутизация:** React Router DOM
*   **Стилизация:** Tailwind CSS
*   **Иконки:** Lucide React
*   **Интернационализация (i18n):** i18next, react-i18next
*   **Бекенд/База данни:** Firebase (Authentication, Firestore, Storage)

## Модел на данните (Firestore - на базата на първоначалния SQL дизайн)

Архитектурата на данните трябва да бъде адаптирана от първоначалния релационен SQL модел към NoSQL модела на Firestore.

### Колекции (Предложени)

1.  **`users`** (Имплементирана)
    *   Съхранява профилна информация за потребителите (uid, name, email, role, createdAt).
    *   Служи като разширение на Firebase Auth за запазване на специфичните роли.
2.  **`recipes`**
    *   Съдържа детайли за рецептите (време за приготвяне, порции, стъпки).
    *   *Под-колекции:* `translations` (за различни езици), `ratings` (оценки).
3.  **`ingredients`**
    *   База с данни за всички възможни съставки (калории, макронутриенти).
    *   *Под-колекции:* `translations`.
4.  **`user_pantry` (или `user_inventory`)**
    *   Запазва наличностите на всеки потребител (съставка, количество, срок на годност).
5.  **`shopping_lists`**
    *   Списъци за пазаруване, генерирани от потребителите.
6.  **`meal_plans`**
    *   Планирани хранения от потребителите.
7.  **`measurements`** (Имплементирана)
    *   Списък с мерни единици (name_bg, name_en, type, base_weight_grams, is_system_unit).
8.  **`activity_logs`** (Имплементирана)
    *   Дневник на всички действия (userId, userEmail, action, details, timestamp).

### Firebase Storage Структура
*   `profiles/{uid}/avatar_{timestamp}.jpg`: Профилни снимки на потребителите.

## Основни модули и интеграции

*   **Автентикация и Роли:** `AuthContext` управлява 4 потребителски роли: Гост, Потребител, Супер потребител и Администратор. Интеграцията се осъществява чрез реална **Firebase Authentication (Email/Password)** и Firestore.
*   **Интернационализация:** `i18next` управлява преводите. Преводите на интерфейса се зареждат локално, докато съдържанието генерирано от потребители (рецепти, съставки) се извлича от базата данни (Firestore).
*   **Стилизация:** Tailwind CSS отговаря за "тъмната" и елегантна тема, базирана на насоките от Stitch.

## Модел на потока на данните (Data Flow)

1.  Потребителят взаимодейства с UI (React компоненти в `src/pages/`).
2.  Компонентите използват React Context (от `src/context/`) или локален State за управление на състоянието.
3.  За мрежови заявки и работа с базата данни се използват функции/сървиси, които комуникират с Firebase API (конфигуриран в `src/lib/firebase.js`).
4.  Промените в базата данни (Firestore) се отразяват обратно в UI чрез слушатели в реално време (Real-time listeners) или при повторно извличане на данните.
